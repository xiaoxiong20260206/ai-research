Anthropic 产品速度密码

40天30+功能、模型吃掉你的产品、PM与工程师融合——解读 Claude Code 产品负责人 Kat Wu 的产品观

📌 信息源：Lenny's Podcast × Kat Wu 🏷 主题：AI产品 · 研发效能 · 组织设计 ⏱ 阅读时间：约15分钟
01 · 研究概述

为什么值得研究这个话题？

Anthropic 在近40天内发布了超过30个功能，几乎每天都有新功能上线。这个速度让整个行业瞩目。在 AI 能力趋同、基础模型越来越接近的时代，产品迭代速度成为真正的竞争护城河。

Lenny Rachitsky 访谈了 Claude Code 和 Cowork 产品负责人 Kat Wu，深度揭秘了这背后的秘密。本报告基于访谈内容的两篇深度解读文章，系统整理核心观点和洞见。

研究焦点：Anthropic 的产品迭代速度如何实现？AI 时代 PM 角色如何演化？如何在模型快速升级的背景下做出经得住时间考验的产品决策？
👤
Kat Wu — Claude Code & Cowork 产品负责人
前多年工程师背景 → 短暂做过 VC → 加入 Anthropic。与 Claude Code 创造者 Boris Cherny 搭档：Boris 定义产品愿景（3-6个月后产品应该长什么样），Kat 把愿景拆解成可执行路径并协调各团队。团队特点：几乎所有 PM 都有工程背景，设计师也曾是前端工程师。

03 · 速度密码

Anthropic 三板斧：流程 × 文化 × 工具

Lenny 追问：是不是因为内部用了 Mythos 模型才这么快？Kat 的回答直接：大部分加速来自流程和团队文化，模型只加快了一点点。

1
目标锁定 — 对抗 LLM 的通用性陷阱
LLM 太通用了，如果不锁定用户和场景，团队很容易迷失方向。Claude Code 的目标用户是专业开发者，某个功能解决的问题是"权限弹窗太多导致疲劳"，目标是让企业开发者能安全地实现零权限提示。目标清晰 → 工程师自主决策，不需要等 PM 拍板。
2
Research Preview 机制 — 降低完美主义门槛
几乎所有功能都以 Research Preview 形式先上线。用户知道这是早期版本，可能不会永久保留。不需要做到完美才发布，1-2 周就能把东西推出去，然后根据真实用户反馈迭代。
3
Evergreen Launch Room — 零摩擦发布流程
工程师觉得功能差不多了，把它丢进 evergreen launch room 频道。Docs 负责人 Sarah、PMM 负责人 Alex、DevRel 团队迅速跟进，第二天就能发布。目标：移除一切阻碍发布的障碍，让每个人都能在一周甚至一天内把想法变成产品。
Kat 原话："我们希望移除一切阻碍发布的障碍。团队里每个人都应该能在一周之内，甚至一天之内，把自己的想法变成产品。"

3.1 模型会"吃掉"你的产品功能

每次新模型出来，Kat 们做的第一件事是——删功能。这是 AI 产品开发中最反直觉却最重要的规律。

🔧
早期模型阶段：产品需要"拐杖"
早期模型做大规模代码重构时，总是改了 5 个调用点就停了，明明有 20 个要改。Claude Code 团队设计了 To-do List 功能，让模型先列清单再逐个完成。这个功能非常管用。
必要拐杖
🚀
Opus 4 之后：拐杖变成累赘
模型自己就会主动列清单、逐个完成了。To-do List 从"必要的拐杖"变成了"可有可无的辅助界面"，反而增加了界面复杂度。
功能退场
✅
Opus 4.5/4.6：代码审查真正可用
代码审查功能试了好几个版本准确率一直不够。直到 Opus 4.5/4.6，才达到了工程团队真正依赖它的水平。现在 Anthropic 内部合并 PR 之前，必须先过 Claude 的代码审查。
模型解锁
系统性做法：每次发布新模型，通读整个 system prompt，逐段反思：模型还需要这个提醒吗？如果不需要了，就删掉。

Mike Krieger 的呼应："为未来的模型做产品。" — 提前做好还没完全能用的原型，新模型一出立即测试是否能弥合。
04 · PM 的未来

PM 角色融合：代码便宜了，判断力变贵了

Kat 面试了几百个想进 Anthropic 做 PM 的人，发现大多数人思维方式不对——还在用旧时代的思路：做多季度路线图、跨团队协调、需求对齐所有相关方。

❌ 旧时代 PM
多季度路线图规划
跨团队协调需求
每个需求对齐所有相关方
充当工程与业务桥梁
功能立项—开发—发布全流程管理
→
✅ AI时代产品人
快速判断哪些值得做
设定清晰目标让工程师自主
搭建低摩擦发布框架
从用户反馈到发布全自主
有产品品味的工程师直接上
💡
产品品味是终极稀缺能力
代码变得越来越便宜。那什么变得更有价值呢？决定写什么。面对 GitHub 数万条 Feature Request，判断哪些值得做、用什么方式做的能力，来自任何背景，但极其稀缺。Anthropic 几乎会无条件录用强烈展示出这种能力的人。
⚡
融合的代价：产品一致性
同一个需求有两个方案都在做，干脆都上线让用户投票。/powerup 功能违背了"产品应该直觉到不需要教程"的原则——但功能太多了，用户确实需要有人告诉他们哪10个是必须用的。
05 · 最被低估的AI技能

让模型反思自己的错误

当模型做了出乎意料的操作（如修改了前端代码但没验证 UI），正确做法不是直接修改 system prompt，而是先问模型为什么这么做。

❌
错误做法
直接修改 system prompt，凭猜测添加规则。结果：prompt 越来越长，但根因未解决。
✅
正确做法
问模型："你为什么不检查 UI？" 模型的回答揭示根因，直接指向 harness 层面的改进方向。

模型的回答往往出人意料，例如：

  • 🔹 "system prompt 里的某段话让我产生了困惑"
  • 🔹 "我把验证任务委派给了 sub-agent，但 sub-agent 没做，而我也没有检查它的工作"
  • 🔹 "我没意识到验证前端是这个任务的一部分"
Kat 总结："对模型的决策保持好奇心，问它为什么做出了那个选择，你就能看到是什么误导了它，然后修复 harness 来填补这个空隙。"
05.2 · 构建 Evals 是最被忽视的工作

大多数人愿意花时间建新功能，但没有人愿意写评估。Kat 认为这是严重误解：

没有 Evals，你就不知道：新功能是否真的有效 / 新模型是否让产品变好了 / 哪里在退步。写好 Evals 本身是一门手艺，需要真正理解产品的目标和失败模式。
06 · 自动化哲学

95% 的自动化 ≈ 没有自动化

很多人把自动化做到 90-95% 准确率就停止投入，觉得差不多了。Kat 的判断恰好相反：

核心观点
95% 自动化 vs 100% 自动化：质变而非量变

一个不能 100% 准确运行的自动化，你就不能真正依赖它。你反而需要持续监督它，这本身就消耗了时间。95% 的自动化不是省了 95% 的力气，而是你既没有省力，又增加了监督负担。

类比：就像自动驾驶 L2 vs L5 — L2（95%）你还得时刻准备接管，L5（100%）你可以睡觉。中间只差 5%，价值差 10 倍。

正确的自动化路径

1
找到真正想自动化的重复性工作
注意你每天在重复做什么。这些重复性工作才是应该优先自动化的。
2
打磨到 100% 可靠
花时间教模型你的偏好，给反馈，让它把反馈固化进 Skills。最后那 5% 确实需要更多时间，但必须投入这个精力。
3
解放时间做真正有价值的事
省下来的时间，去做真正有创造力的部分。这才是 AI 给每个人的真正杠杆。
07 · 本质洞察

透过现象看本质

洞察 1 · 速度本质
速度 = 决策权下放 × 零摩擦流程
当工程师不需要等 PM 批准、不需要跨部门会议、不需要完美才发布，当目标足够清晰让每个人都能自主决策，系统吞吐量自然暴涨。速度不来自执行更猛，而来自把路修顺了。
类比：就像高速公路不是让车跑得更快，而是去掉所有红绿灯。Anthropic 的速度来自"把路修顺了"，而非"踩油门更猛"。
洞察 2 · 产品设计
AI 产品设计 = 动态减法
传统产品是加法游戏（不断添加功能），AI 产品是减法游戏（不断删除模型已能自己做的功能）。产品的"辅助轮"会随模型成长而退场。未来产品 UI 会越来越简化，很多今天的界面元素会消失。
类比：像养孩子——小时候需要辅助轮的自行车，长大了辅助轮反而成了累赘。产品团队的核心工作将变成"持续判断哪些功能模型已经能自己做"。
洞察 3 · 产品品味
产品品味 = 在无限可能中做减法的能力
当技术实现成本趋近于零，稀缺资源从"能不能做"变成"该不该做"。判断力成为最稀缺的生产要素。工程师与 PM 的界限会继续模糊，最终合并为"Product Engineer"，但"产品品味"不会消失，只是不再绑定特定职位。
类比：就像米其林餐厅的价值不在于"会做菜"，而在于"知道只做哪8道菜"。
洞察 4 · 组织文化
使命对齐 = 天然决策框架
Anthropic 没有做社交网络、没有做资讯 Feed，没有做和主线任务无关的产品，即使可能赚钱也不做。使命提供了一把筛子，把所有可能分散注意力的方向都过滤掉，让执行密度极高。
Kat 的极端表态："如果 Claude Code 失败了但 Anthropic 整体成功，我会很开心。" — 这是真正使命驱动的团队才有的思维方式。
08 · 趋势判断与行动建议

从现在到未来

📈
趋势 1：从对话式 → 行动式 AI
2024年前：AI 是对话工具（告诉你怎么做）。2025年起：AI 是行动智能体（替你做完）。当用户第一次感受到智能体真的替自己完成了一件事，对 AI 的判断会彻底改变。

行动：开始构建"能每天用"的 AI 工作流，而非停留在原型阶段。
🔀
趋势 2：并行智能体时代
阶段1→单任务成功 / 阶段2→6个并行（当前）/ 阶段3→50-100个并行（近未来）。届时任务需要跑在云端，模型能自我验证、自我改进。

行动：提前适应"管理多个 AI 智能体"的工作模式，学习设计并行工作流。
🎯
趋势 3：产品品味成为终极稀缺
随着 AI 能力趋同，产品判断力成为真正的护城河。能展示这个能力的人，会成为组织的核心资产。

行动：工程师培养产品直觉；PM 要么学写代码，要么深化产品品味到不可替代级别。

能力要求分布图（预测）

产品品味 / 判断力 ↑ 价值暴涨
Evals / 评估体系构建 ↑ 越来越重要
工程背景 / 代码能力 → 持续重要
传统 PM 协调能力 ↓ 相对贬值
纯执行 / 机械重复 ↓↓ 被自动化
